# コードレビューワークフロー

## 使い方

```bash
/review [ベースブランチ名]
```

- 引数なし: デフォルトで `main` ブランチとの差分を確認してレビュー
- 引数あり: 指定されたブランチ（例: develop）との差分を確認してレビュー

## 実行手順

以下の段階的な手順に沿ってコードレビューを実施します。

### 1. プロジェクトのルール確認

- 最初に必ず `GEMINI.md` を読み込んでください。
- `GEMINI.md` 内で参照されているコーディング規約やアーキテクチャルールなどの関連ファイルも確認し、プロジェクトのルールを把握してください。

### 2. 差分の抽出

- コマンドの引数で渡されたブランチ（指定がない場合はデフォルトの `main`）として、現在のブランチ（HEAD）との差分を抽出します。
- 以下のコマンドを利用して差分を確認してください。
  ```bash
  # 変更されたファイルの一覧を取得
  git diff {ベースブランチ名}...HEAD --name-only
  # 実際の差分を取得
  git diff {ベースブランチ名}...HEAD
  ```

### 3. 対象ファイルの読み込みと分析

- 差分が含まれるファイルのうち、レビュー対象となるファイル全体を読み込んでください。（例: view_fileを使用）
- 追加・変更された実装の詳細とコンテキストを把握します。

### 4. コードレビュー（観点）

抽出した差分および対象ファイルについて、以下の観点でレビューを行ってください。

#### A. 規約準拠チェック

- **ルールの遵守**: `GEMINI.md`経由で読み込んだすべてのルールファイル（コーディング規約、アーキテクチャなど）に対して、コードが遵守しているか確認してください。
- **コメントと実装の整合性**: TSDocなどのコメントと実際のコードの挙動が乖離していないか確認してください。
- **仕様と実装の乖離**: 変更内容が仕様（仕様書が存在する場合）と乖離していないか確認してください。

#### B. コード品質チェック

- **ロジックの正しさ**: エッジケースが考慮されているか、条件分岐が網羅されているか。
- **エラーハンドリング**: API呼び出しやユーザー入力に対して適切なエラー処理（try-catch、フォールバックなど）が行われているか。
- **型安全性**: `any` 型の不必要な使用がないか、型の定義が適切に行われているか。
- **パフォーマンス**: 不要な再レンダリングの原因となる実装がないか、メモ化（`useMemo`, `useCallback`など）の欠如がないか、不適切なデータフェッチが行われていないか。

### 5. レビュー結果の出力

レビューが完了したら、以下の形式に従って重要度別に分類して結果を出力してください。
該当する指摘がないカテゴリは省略し、全体を通して指摘がゼロの場合は「問題は見つかりませんでした」と報告してください。

```markdown
## レビュー結果

### 🔴 Error（修正必須）

- [ファイルパス:行番号] 指摘内容と修正提案

### 🟡 Warning（修正推奨）

- [ファイルパス:行番号] 指摘内容と修正提案

### 🔵 Info（参考情報）

- [ファイルパス:行番号] 指摘内容と改善提案

### ✅ Good Points

- 良かった点があれば記載
```
