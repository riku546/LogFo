# 活動サマリー生成機能 詳細設計書 (Activity Summary Design)

## 1. フロントエンド実装設計 (Frontend)

### 1.1. コンポーネント構成 (`src/features/summary/`)

活動ログと外部データをAIに食わせて要約・自己PRを生成し、加筆修正する画面です。

- **`SummaryGeneratorPage`**: メインの管理画面。生成された過去のサマリー一覧と新規生成機能を持つ。
- **`components/SummaryConfigPanel`**:
  生成するためのコンテキスト設定エリア。対象期間（例:「過去半年」や「特定のロードマップID」）を選択するドロップダウンコンポーネント。
- **`components/SummaryChatStream`**:
  ロードマップ生成時と同様の、ストリーミングでプレビュー表示部分にテキストが徐々に出力されるUI。Vercel AI SDK の `useCompletion` などを使用。
- **`components/SummaryEditor`**:
  生成が完了した、またはDBから読み込んだサマリーテキストを手動で調整（Markdown加筆）し、保存するためのテキストエリアまたはリッチエディタ。

### 1.2. 状態管理 (State Management)

- **AI生成中/完了状態 (Vercel AI SDK):** ストリーミング開始から完了までのステート。
- **Editor 状態 (useState / React Query Mutation):** ユーザーが手直しした最終的なテキストをステートに持ち、「保存」ボタンのクリックと共にDB（`Hono RPC` 経由）へ `POST / PUT` する。

## 2. バックエンド・API設計 (Backend)

### 2.1. API エンドポイント (Hono RPC)

#### ① サマリー生成 (Streaming API)

- **エンドポイント:** `POST /api/summary/generate`
- **処理フロー:**
  1.  リクエストに含まれる検索条件（ロードマップIDや対象期間）をもとに、Drizzle ORM で `activity_logs` と `external_activities`（該当期間のサマリー等）を取得する。
  2.  取得した複数レコードを加工し、テキスト上の情報としてフォーマットする（例: `[2024-03-01: GitHub 5 Commits, WakaTime 2hrs, Log: React Hooksについて学習]`）。
  3.  フォーマットした活動コンテキストをシステムプロンプトに結合し、「この活動記録をもとに、『どのような壁にぶつかり、どう解決し成長したか』という一貫したストーリーを含む自己PR文・振り返りレポートを作成せよ」と指定。
  4.  Vercel AI SDK と OpenRouter経由の `@openrouter/glm-4.5-air` で `streamText` の出力をフロントエンドへSSEで返却する。

#### ② サマリー保存 (CRUD API)

- **エンドポイント:** `POST /api/summary`
- **処理フロー:** サマリー生成完了後、もしくはユーザーが手動編集を終えた際に呼び出され、`summaries` テーブルへマークダウンテキストとメタ情報（タイトルや期間情報など）を保存する。

### 2.2. AI プロンプト構築戦略 (Prompt Engineering)

大量の活動記録が数ヶ月分たまっている場合、そのまま文字列結合するとLLMの入力（コンテキスト長）制限に達する、またはノイズが多くなり良いサマリーが出力されない懸念があります。

- **データ整形策:** 毎日の細かいログはWorker側で「週次また月次のトピック抽出」として軽くまとめ直すか、あるいはシステムプロンプト内で「以下のログ内の『重要な学習内容』『苦労した点』に関わる記述を重点的に拾い上げてサマリーを構成すること」などの具体的なフィルタリング指示を与えます。

## 3. エラーハンドリング・コーナーケース

1.  **AIプロンプト長超過:** 文字列に結合した結果、トークン制限（Hono側でのPayloadサイズ等）を超える規模になりそうな場合は、ログの先頭から文字数制限でトリミングするか、最新のレコードN件に絞り込んでからLLMに投げる処理を挟みます。
