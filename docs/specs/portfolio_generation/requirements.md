# ポートフォリオAI文章生成・共有機能 要件定義書 (Portfolio AI Narrative Requirements)

## 1. 機能概要

本機能は、ユーザーが選択したサマリー・自由入力テキスト・既存の生成文章をもとに、LLMがポートフォリオ向け文章（自己PR・強み・学び・将来）を生成し、公開ページで共有できるようにする機能です。生成UIはサイドバーのチャット形式で提供し、生成結果はストリーミング表示します。

現行の「公開ページでロードマップ/サマリー本文を直接表示する構成」は廃止し、AI生成文章を中心とした構成へ移行します。

## 2. 対象ユーザー

- 就職・転職活動中のエンジニア
- 学習成果を対外的に分かりやすく提示したいユーザー
- 手動で自己PR文章を組み立てる時間を削減したいユーザー

## 3. ユースケース (User flow)

1. ユーザーがビルダー画面（`/portfolio`）を開く。
2. 中央表示エリアを編集モードに切り替え、プロフィール情報（表示名、SNS、経歴、スキル、自己PR等）を直接編集する。
3. サイドバーのAIチャットで、対象項目（自己PR/強み/学び/将来）を選択し、自由入力テキストを入力する。必要に応じて「サマリー選択」ボタンからモーダルを開き、生成入力用サマリー（最大5件）を選択する。
4. チャット経由でストリーミングされた生成結果（単一テキスト）を確認し、ワンクリックで選択中の項目へ反映する。
5. 必要に応じて中央表示エリアで手編集する。
6. 既存の保存ボタンでポートフォリオを保存する。
7. 公開ページ（`/portfolio/{slug}`）で `PR・強み` タブに生成文章を表示し、外部共有する。

## 4. 機能要件 (Functional Requirements)

### 4.1. ポートフォリオ設定・構築機能

- プロフィール設定:
  - 表示名（必須）
  - 自己紹介
  - アバターURL
  - SNSリンク（GitHub, X, Zenn, Qiita, AtCoder, Website）
  - 経歴ストーリー（複数件の追加/編集/削除）
  - スキル（文字列タグの追加/削除）
- 編集UI:
  - 中央表示エリアで、以下を直接編集可能にする
    - 表示名、自己紹介、アバターURL
    - SNSリンク
    - 経歴ストーリー
    - スキル
    - 自己PR、強み、学び、将来
  - 編集方式はページ全体の「閲覧モード / 編集モード」切替とする
  - レイアウトは左サイドバー + 右ペインの2カラムとし、サイドバーは画面最下部までの高さを使用する
- 公開設定:
  - Slug必須、英小文字/数字/ハイフンのみ、最大50文字
  - 公開/非公開切替
  - 保存/公開設定導線は右ペイン下部に配置する

### 4.2. AI文章生成機能

- 生成対象は以下4セクションとする:
  - 自己PR (`selfPr`)
  - 強み (`strengths`)
  - 学び (`learnings`)
  - 将来 (`futureVision`)
- 4セクションはすべて任意表示（空文字の場合は公開ページで非表示）。
- サマリー選択UI（サイドバー）:
  - メッセージ入力欄にはサマリー選択ボタンのみ表示する
  - ボタン押下でサマリー一覧モーダルを表示する
  - ユーザー自身のサマリーから選択可能
  - 選択上限は5件
  - モーダルは「適用」押下時にのみ選択結果を確定し、「キャンセル」では変更を破棄する
- チャットUI:
  - 上部に時系列メッセージ表示、下部に入力欄を配置する
  - 入力欄には対象項目セレクト（`selfPr` / `strengths` / `learnings` / `futureVision`）を設ける
- チャット入力:
  - 自由入力テキスト（必須）
- 生成実行条件:
  - 自由入力テキストが非空であること
- 生成操作:
  - サイドバーAIチャット経由で生成する
  - 応答はSSEによるストリーミング表示とする
- 反映操作:
  - AIの生成候補は単一テキストとする
  - ワンクリック適用時は、セレクトで選択中の項目1件のみ `generatedContent` を上書きする
- 履歴保持:
  - チャット履歴は画面表示中のみ保持し、ページ再読込時は保持しない
- 生成文体:
  - 就活/転職向けフォーマル
  - 各項目 200-350字を目安

### 4.3. 公開表示機能

- 公開URLは `logfo.app/portfolio/{slug}` 形式。
- 公開ページの2ndタブ名は `PR・強み`。
- `PR・強み` タブにはAI生成4セクションのみ表示する。
- 旧ロードマップ/サマリー本文表示は行わない。
- サマリーは生成入力用データとしてのみ利用し、公開本文として表示しない。
- 公開ページ（`/portfolio/{slug}`）は閲覧専用とし、編集UIは提供しない。

### 4.4. データ保存

- 保存対象は `PortfolioSettings` に保持する。
  - `profile`
  - `generation`（`selectedSummaryIds`, `chatInput`, `targetSection`）
  - `generatedContent`（4セクション）
- 生成結果の確定タイミングは既存の保存ボタン押下時とする。

## 5. 受け入れ基準 (Acceptance Criteria)

- 必須入力条件:
  - ポートフォリオ保存時は表示名・Slugが必須
  - 生成API実行時は、自由入力テキスト空を拒否
  - 生成API実行時は、`targetSection` が4セクションのいずれかであること
- 表示条件:
  - 公開ページでは4セクションの非空項目のみ表示
  - 旧ロードマップ/サマリー本文は表示されない
  - 公開ページは閲覧専用である
  - ビルダー画面では左サイドバーが画面最下部まで表示される
  - 保存/公開設定の操作UIは右ペイン下部に表示される
- 操作条件:
  - メッセージ入力欄にはサマリー一覧を直接表示しない
  - サマリーはモーダル内で選択し、適用押下時のみ `selectedSummaryIds` に反映される
  - モーダルでキャンセルした場合、未確定の選択変更は破棄される
- エラー条件:
  - サマリー選択が6件以上の場合はバリデーションエラー
  - ユーザーが所有しないサマリーIDを指定した場合はエラー

## 6. 非機能要件 (Non-Functional Requirements)

- レスポンシブ対応: モバイル/デスクトップで崩れないこと。
- UX:
  - 生成中状態（ストリーミング中）を明示すること。
  - 閲覧モード / 編集モードが明確に識別できること。
  - 生成候補適用時に、どの項目へ反映されるかを確認できること。
  - サマリー件数が増えても入力欄の高さが逼迫しないこと（選択UIはモーダルで完結すること）。
- パフォーマンス:
  - サマリー選択UI用データ取得はN+1を避けた一覧取得APIを使うこと。
