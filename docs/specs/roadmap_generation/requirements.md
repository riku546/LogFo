# ロードマップ生成機能 要件定義書 (Roadmap Generation Requirements)

## 1. 機能概要

ユーザーが自身の現状（スキル、経験など）と目標（志望企業、取得したいスキル・資格など）を入力し、以下2つの方法でロードマップを作成できる機能。

- **AI生成モード:** LLM（Large Language Model）を利用して、目標達成までの具体的な学習ステップ（マイルストーンおよびタスク）を自動生成する。
- **手入力モード:** ユーザーがマイルストーンおよびタスクを直接入力して、ロードマップを作成する。

## 2. 対象ユーザー

- エンジニアを目指す初学者・学生
- 新しい技術スタックやポジション（例: フロントエンドからバックエンドへ）を目指すエンジニア
- 転職活動に向けて、どの企業にどのようなスキルが求められているかを知り、学習計画を立てたいユーザー

## 3. ユースケース (User flow)

1.  **作成方式選択フェーズ:** ユーザーが「AI生成」または「手入力」を選択する。
2.  **入力フェーズ:** ユーザーが「現在地」と「目的地」に関する情報をフォームに入力する。
3.  **分岐フェーズ:**
    - **AI生成モード:** システムが入力情報をもとにLLMへプロンプトを送信し、ロードマップデータを受け取る。
    - **手入力モード:** ユーザーがマイルストーンおよびタスクを直接追加・編集する。
4.  **確認・編集フェーズ:** ユーザーがロードマップを確認し、必要に応じてマイルストーンやタスクを追加・修正・削除する。
5.  **決定フェーズ:** ロードマップを確定し、日々の活動記録（Activity Tracking）のベースとしてシステムに保存する。

## 4. 機能要件 (Functional Requirements)

### 4.1. 情報入力機能 (Frontend)

- ユーザーが以下の情報を入力できるフォームを提供する。
  - **現在の情報:**
    - 現在の職種/学生
    - 経験のあるプログラミング言語・フレームワーク（複数選択可）
    - その他のスキル・資格
    - 1日に学習に割ける平均時間（目安）
  - **目標の情報:**
    - 第一志望の企業名（任意、複数可）
    - 目指す職種/ポジション（例: フロントエンドエンジニア、SREなど）
    - 特定の習得したい技術や資格（任意）
    - 目標達成までの希望期間（例: 6ヶ月後、1年後）
  - **PDFファイルのアップロード:**
    - ユーザーの履歴書や、志望企業の会社説明資料（Speaker DeckなどのPDF）をアップロードしてコンテキストとして分析に利用できる機能を提供する。

### 4.2. ロードマップ生成機能 (Backend)

- ユーザーの入力情報および抽出したPDFのテキストデータを構造化し、LLM（OpenRouterが提供する `glm-4.5-air` モデルを使用）に適したプロンプトを構築する。
- **Web検索による情報補完:**
  - ユーザーが「志望企業名」のみを入力し、詳細な資料をアップロードしていない場合、GLM-4.5-Airのツール呼び出し（Tool Invocation / Web Browsing 機能）、または外部の検索API（Tavily, Firecrawlなど）を併用して、該当企業の最新の採用技術スタックや求める人物像をWeb経由で自動収集し、プロンプトのコンテキストに含める。
- LLMから以下の構造化データ（JSON等）としてロードマップのレスポンスを受け取る。
  - **全体サマリ:** ギャップの解説と、学習方針の簡単な説明。
  - **マイルストーン (複数):** 中間目標（例: 「Reactの基礎を習得する」「モダンなフロントエンド環境を構築する」）。
  - **タスク (マイルストーンごと):** 具体的な行動単位（例: 「React公式チュートリアルを完了する」「Todoアプリを作成する」）。
  - 各タスクの予想学習時間。

### 4.3. 編集・カスタマイズ機能 (Frontend)

- 生成されたマイルストーンとタスクをツリー/かんばん形式等で視覚的に表示する。
- ユーザーのアクション：
  - マイルストーン/タスクのテキスト編集。
  - 新規マイルストーン/タスクの追加。
  - ドラッグ＆ドロップによる順序の入れ替え。
  - 不要なタスクの削除。

### 4.4. 手入力ロードマップ作成機能 (Frontend)

- ロードマップ一覧の「新しいロードマップ」導線で、`AI生成` と `手入力` の作成方式を選択できる。
- 手入力モードで以下を入力・編集できる。
  - **ロードマップ基本情報**
    - 現在の状態 (`currentState`) ※必須
    - 目標の状態 (`goalState`) ※必須
    - サマリー (`summary`) ※任意
  - **ロードマップ構造**
    - マイルストーンの追加・編集・削除
    - タスクの追加・編集・削除
    - タスクの予想学習時間（任意入力）
- 手入力保存時の最小条件:
  - マイルストーンが1件以上
  - 全体でタスクが1件以上
  - マイルストーン名およびタスク名が空でない

### 4.5. ロードマップ保存・管理機能 (Backend)

- 確定されたロードマップの構造（マイルストーン、タスク、順序）をデータベースに保存する。
- 保存対象は **AI生成モード** と **手入力モード** の両方を含む（生成経由でなくても保存可能）。
- 後からロードマップを再生成、または部分的に再生成（特定のタスクのみ展開など）できる機能（オプション）。

## 5. 非機能要件 (Non-Functional Requirements)

- **レスポンスタイム:** LLMの生成には時間がかかるため、フロントエンドでは適切なローディングUI（スケルトンスクリーンや「生成中...」のアニメーション）を表示し、タイムアウトエラーをハンドリングする。
- **データフォーマット:** LLMからの出力がパースエラーにならないよう、`Structured Outputs` などの機能を用いて厳密なJSON出力を強制する。
- **コストおよび制限:** LLM APIの呼び出しコストを抑えるため、1ユーザーあたりの生成回数制限（Rate Limiting）を設ける。

## 6. 特筆事項 / 検討課題

- 大規模なロードマップになるとタスクが多すぎてユーザーが圧倒される可能性があるため、最初に見せる粒度や、進捗に合わせてタスクを展開（順次生成）するアプローチも検討する。
