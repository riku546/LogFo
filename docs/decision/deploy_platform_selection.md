# Architecture Decision Record: デプロイ環境としてのCloudflare Workers採用

**タイトル:** デプロイ環境としてのCloudflare Workers採用
**ステータス:** Accepted

### Context (背景と要件)

個人開発プロジェクトのデプロイ環境を選定するにあたり、コスト、デプロイ容易性、スケーラビリティ（水平）、開発体験、コンピューティングサービスの豊富さという5つの観点から、Cloudflare Workers、Render、Railwayの3つのプラットフォームを比較検討しました。

プロジェクトの要件として、以下の観点を重視しました：

- **コスト**: 個人開発のため、初期コストを抑えつつ、スケール時のコスト予測可能性が重要
- **デプロイ容易性**: TypeScriptプロジェクトの迅速なデプロイと自動化
- **スケーラビリティ**: 将来的なトラフィック増加への対応力
- **開発体験**: TypeScript開発環境との親和性、開発効率
- **コンピューティングサービスの豊富さ**: データベースやストレージなど必要なサービスの利用可能性

### Options Considered (検討した選択肢)

TypeScriptとの親和性が高いプラットフォームとして、以下の3つを候補に選定：

- **Cloudflare Workers**: エッジコンピューティングプラットフォーム
- **Render**: フルマネージドPaaS
- **Railway**: 開発者体験重視のPaaS

RenderとRailwayも優れたプラットフォームですが、グローバルスケーラビリティ、コストパフォーマンス、サービスの豊富さの観点から、Cloudflare Workersが最適な選択と結論づけました。

### Decision (決定事項)

## 最終選定：Cloudflare Workers

上記の比較検討の結果、Cloudflare Workersを選定しました。その決定的な理由は以下の通りです：

### 1. グローバルスケーラビリティの圧倒的優位性

300ヶ所以上のエッジロケーション（2025年時点で330+都市）での自動分散実行により、世界中のどこからアクセスしても一貫した低レイテンシを実現できます。コールドスタートが存在しないため、初回アクセスでも即座にアプリケーションが起動し、ユーザー体験を損ないません。

将来的にユーザーベースがグローバルに拡大した場合でも、インフラストラクチャの変更なしに対応できる点は、長期的な視点で大きなメリットです。

### 2. コストパフォーマンスの高さ

無料プランで1日あたり10万リクエスト（月間約300万リクエスト）まで対応できるため、プロジェクトの初期段階では完全に無料で運用できます。有料プラン（月額$5）では、リクエストは$0.30/million、CPU時間は$0.02/million CPU msという従量課金制となっており、実際の使用量に応じた明確な課金で、大半の個人プロジェクトを低コストでカバーできます。

従量課金制により、実際の使用量に応じた明確な課金となるため、予期せぬ高額請求のリスクが低く、コスト管理が容易です。

### 3. TypeScript開発体験の優秀性

完全な型サポート、自動型生成、IntelliSenseの完璧な動作により、TypeScript開発における生産性が非常に高くなります。型と実際のランタイム動作の完全な同期により、開発時の信頼性も確保されています。

### 4. 豊富なコンピューティングサービス

D1、KV、R2、Durable Objects、Queues、Workers AI、Vectorizeなど、現代的なアプリケーション開発に必要なサービスが包括的に提供されています。これらがすべてBindingsという統一的なAPIでアクセスでき、サービス間の連携が容易です。

将来的なアプリケーション拡張時にも、必要なサービスがエコシステム内に揃っているため、他のプラットフォームへの移行を検討する必要がありません。

### 5. 将来性への投資

Cloudflareは、エッジコンピューティング分野のリーディングカンパニーとして、継続的に新機能とサービスを追加しています。Workers AI、Vectorize、Workflowsなど、最新技術への対応が早く、将来的な技術トレンドに追従できる可能性が高いです。

### Consequences (結果・影響)

Cloudflare Workersを選定するにあたり、以下の技術的制約を認識し、対処方法を事前に検討しておく必要があります。

### 1. 長時間処理の制約

Workersは短時間のリクエスト処理に最適化されており、CPU時間が30秒（デフォルト、最大5分まで設定可能）に制限されています。データ集計、大量のデータ処理、複雑な計算など、時間のかかる処理には不向きです。

**対処方法：**
- Workers Queuesを使用して非同期処理に移行し、タスクを分割して並列実行
- Durable Objectsで状態を保持しながら、複数のリクエストに分けて処理
- 重い処理はCloudflare Workflowsで複数ステップに分割
- 必要に応じて外部のバッチ処理システム（AWS Lambda、Google Cloud Functionsなど）と連携

### 2. D1データベースのスケール制約

D1はSQLiteベースで、以下の制限があります：
- 1データベースあたり最大10GB
- アカウントあたり最大50,000データベース
- ToC（対消費者）プロダクトでユーザー数が50,000を超える場合、データベース総数の上限に到達するリスクがあります。

**対処方法：**
- **マルチテナンシー設計**：複数ユーザーを1つのD1データベースに収容し、テナントIDで分離
- **シャーディング戦略**：ユーザーIDのハッシュ値に基づいて複数データベースに分散（例：100ユーザー/DB = 最大500万ユーザー対応）
- **課金プランごとに変更**：無料プランでは複数人のDBを割り当て、課金プランでは単一のDBを割り当てるなど。

### 3. データベーススケール時の移行戦略

50,000ユーザーを超えた場合の具体的な移行パス：

- **段階1（〜10万ユーザー）**：D1マルチテナンシー（1DB = 100〜1,000ユーザー）
- **段階2（10万〜50万ユーザー）**：D1シャーディング + 読み取りレプリカ活用
- **段階3（50万ユーザー超）**：Hyperdrive + 外部PostgreSQL（Neon Serverless、Supabase）
- **段階4（数百万ユーザー）**：完全な外部DB移行 + Cloudflare WorkersはAPIレイヤーとして継続使用

これらの制約は、Cloudflare Workersが「エッジファースト」のアーキテクチャであることに起因しますが、適切な設計と段階的な移行計画により、初期段階から大規模化まで対応可能です。

## 各プラットフォームの評価

### Cloudflare Workersの強み

Cloudflare Workersは、以下の点で他のプラットフォームを大きく上回っていました。

**グローバルスケーラビリティ**

世界275ヶ所以上のデータセンターでアプリケーションが自動的に実行され、ユーザーに最も近い場所からコンテンツを配信できます。この分散アーキテクチャにより、地理的に離れたユーザーでも一貫して低レイテンシでアクセスできる点が、他のプラットフォームと比較して圧倒的な優位性です。

また、V8 Isolate技術によりコールドスタートが存在せず、常に即座にアプリケーションが起動します。これはサーバーレスプラットフォームにありがちな初回アクセス時の遅延を完全に排除し、ユーザー体験を損ないません。

**優れたコストパフォーマンス**

無料プランでも1日あたり10万リクエスト（月間約300万リクエスト）まで対応できるため、個人開発プロジェクトの初期段階では完全に無料で運用可能です。有料プラン（月額$5基本料金）では、リクエストは$0.30/million、CPU時間は$0.02/million CPU msの従量課金制となっており、多くの個人プロジェクトを低コストでカバーできます。

特に重要なのは、従量課金制により使用量に応じた明確な課金体系である点です。リクエスト数とCPU時間に基づく課金モデルは予測可能性が高く、予期せぬコストの発生を防げます。また、R2ストレージを使用すれば、通常のクラウドプロバイダーで高額になりがちなエグレス（データ転送）費用が完全に無料です。

**TypeScript完全対応**

Cloudflare Workersは、TypeScript開発において最高レベルのサポートを提供しています。`wrangler types`コマンドにより、プロジェクトの互換性日付と設定に基づいた正確な型定義が自動生成されます。これにより、使用可能なAPIが正確に型付けされ、IntelliSenseによるコード補完が完璧に機能します。

workerdランタイムから直接生成される型定義は、公式のランタイム実装と完全に同期しているため、型と実際のランタイム動作の乖離がありません。この点は、開発時の生産性とコードの信頼性を大きく向上させます。

**豊富なコンピューティングサービス**

Cloudflare Workersエコシステムは、現代的なアプリケーション開発に必要なサービスを包括的に提供しています：

- **D1**: サーバーレスSQLデータベース。SQLiteベースで、グローバル読み取りレプリケーション対応
- **Workers KV**: 低レイテンシKey-Valueストア。エッジキャッシングに最適
- **R2**: オブジェクトストレージ。S3互換でエグレス費用無料
- **Durable Objects**: ステートフルストレージとリアルタイム調整
- **Queues**: 保証配信のメッセージキュー
- **Workers AI**: サーバーレスGPUによる機械学習モデル実行
- **Vectorize**: AI用ベクトルデータベース

これらのサービスはすべてBindingsという統一的なAPIでアクセスでき、サービス間の連携が容易です。また、すべてのサービスがエッジで実行されるため、一貫した低レイテンシを維持できます。

**デプロイの容易性**

Wrangler CLIを使用することで、`wrangler deploy`コマンド一つでグローバルデプロイが完了します。GitHub Actionsとの統合も簡単で、プッシュ時の自動デプロイを数行の設定で実現できます。

ローカル開発環境も充実しており、`wrangler dev`でローカルサーバーを起動でき、コード変更時の自動リロードにも対応しています。本番環境とほぼ同じ環境でローカルテストできるため、デプロイ後の予期せぬ動作を最小限に抑えられます。

### Renderの評価と不採用理由

Renderは、従来型のPaaSとして優れたプラットフォームですが、いくつかの制約から今回のプロジェクトには不適と判断しました。

**スケーラビリティの制約**

Renderの最大の問題は、スケーラビリティの制約です。無料プランとHobbyプラン（$7/月〜）では、オートスケーリング機能が利用できません。Professionalプラン（$85/月〜）以上でないと、トラフィック増加に応じた自動スケーリングができないため、急激なアクセス増加に対応できません。

また、Renderのドキュメントではリージョン（データセンター拠点）の具体的な数や場所が明示されていませんが、米国とヨーロッパを中心とした限定的な拠点展開となっており、グローバル展開を考えた場合、アジアやオセアニアのユーザーに対するレイテンシが課題となる可能性があります。

**コストの不透明性と高額化のリスク**

Renderの料金体系は複雑で、リソース使用量に加えて、帯域幅、ビルドパイプライン時間、チームメンバー数など複数の課金要素があります。スケール時のコストを正確に予測することが困難で、予期せぬ高額請求のリスクがあります。

特に、Professionalプランへの移行が必要になった場合、基本料金が$85/月と高額になり、個人開発プロジェクトの予算を大きく圧迫します。

**コンピューティングサービスの限定性**

Renderが提供するマネージドサービスは、PostgreSQL、Redis、Key-Valueストアなど基本的なものに限られます。Workers AIやVectorizeのような最新のAI関連サービスや、Durable Objectsのようなステートフル処理に特化したサービスは提供されていません。

将来的にアプリケーションを拡張する際、必要なサービスが不足する可能性が高く、拡張性に課題があります。

### Railwayの評価と不採用理由

Railwayは、優れた開発者体験を提供するプラットフォームですが、グローバル展開とスケーラビリティの面で課題があります。

**グローバル展開の弱さ**

Railwayのデータセンターは4拠点のみで、エッジコンピューティングに対応していません。これは、世界中のユーザーに対して一貫した低レイテンシを提供することが困難であることを意味します。

現代のWebアプリケーションでは、グローバルなユーザーベースを想定することが一般的ですが、Railwayのアーキテクチャでは地理的に離れたユーザーへのパフォーマンスが制限されます。

**スケーラビリティの限界**

Railwayの水平スケーリングは、レプリカ機能により最大10レプリカまでサポートされていますが、これは大規模トラフィックには不十分です。また、リージョン選択の柔軟性が低く、トラフィックパターンに応じた最適な配置が困難です。

Cloudflare Workersのような無制限に近いスケーラビリティと比較すると、将来的な成長の余地が限定されます。

**コスト面の懸念**

Railwayは使用量ベースの課金モデルを採用していますが、実際のCPU・メモリ使用率に基づく課金のため、トラフィック増加時のコスト予測が困難です。クレジット制を採用しており、含まれるクレジットを超過すると追加請求が発生しますが、この超過タイミングの予測が難しく、コスト管理の不確実性があります。

また、無料プランが廃止されており、最低でも月額$5が必要です。Cloudflare Workersの充実した無料プランと比較すると、初期コストの面で不利です。

**エッジコンピューティング非対応**

Railwayは従来型のサーバー配置モデルを採用しており、エッジコンピューティングには対応していません。これは、ユーザーに近い場所でコードを実行し、レイテンシを最小化する現代的なアーキテクチャパターンを採用できないことを意味します。

グローバルなサービス提供を考えた場合、この制約は大きなデメリットとなります。

## 結論

個人開発プロジェクトのデプロイ環境として、Cloudflare Workersを採用することで、以下のメリットが得られると判断しました：

1. グローバルユーザーに対する一貫した低レイテンシの実現
2. 初期コストを抑えつつ、スケール時のコスト予測可能性を確保
3. TypeScriptによる生産性の高い開発体験
4. 将来的なアプリケーション拡張に必要なサービスの包括的な提供
5. 最新技術トレンドへの追従可能性

RenderとRailwayも優れたプラットフォームですが、グローバルスケーラビリティ、コストパフォーマンス、サービスの豊富さの観点から、Cloudflare Workersが最適な選択と結論づけました。

この選定により、プロジェクトの初期段階から成長期、さらには大規模化まで、一貫したプラットフォーム上で開発を継続でき、長期的な技術的負債を最小限に抑えられると考えています。
