# Architecture Decision Record: フロントエンドアーキテクチャの選定（軽量 Feature-Driven 構成）

**タイトル:** フロントエンドアーキテクチャへの軽量 Feature-Driven 構成の採用
**ステータス:** Accepted

### Context (背景と要件)

LogFoのフロントエンドは、ユーザー体験（UI/UX）の質を最重要視し、滑らかなストリーミング表示や直感的な操作感を実現する必要がある。
アーキテクチャ選定の基本的な評価軸はバックエンドと同様（スピード、テスト容易性、学習/オンボーディングコスト）だが、フロントエンドには以下の**限定的な要件と制約**が存在する。

1. **スピード (UI/UXへの注力):** 滑らかなアニメーションや使い心地の良いUIの構築に最大のリソースを割くため、アーキテクチャ自体の複雑さによる開発の遅延は避けたい。
2. **テスト容易性と制約:** SDD（テスト駆動のAIコーディング）による品質担保を目指すが、**開発者自身がフロントエンドの単体テスト（Jest/Vitest/React Testing Library等）の実装経験がない**という強い制約がある。そのため、「テスト対象がシンプルかつ明確に分離されていること」が絶対条件となる。
3. **学習・オンボーディングコスト:** バックエンド側で学習曲線の急な「オニオンアーキテクチャ」という重い選択（自己投資）を行ったため、プロジェクト全体の進行バランスを鑑み、フロントエンド側の構造は直感的でキャッチアップしやすいものに留めたい。

### Options Considered (検討した選択肢)

1. **Feature-Sliced Design (FSD)**
   - **概要:** App, Pages, Widgets, Features, Entities, Shared という厳密な6階層のレイヤーによって依存関係を一方向に強制する強力なアーキテクチャ。
   - **Pros:** チームが巨大になった際のスケール耐性が最強で、スパゲッティコードを完全に防げる。
   - **Cons:** 「このUIはWidgetかFeatureか？」といった分類への迷いが頻繁に発生し、初期の開発スピードが著しく落ちる。また、フロントエンドテスト未経験の状態でこの厳密なルールを適用することは学習負荷が高すぎ、本来注力すべきUI/UXの作り込みからリソースを奪ってしまう。

2. **軽量 Feature-Driven (Bulletproof React スタイル)**
   - **概要:** アプリケーション全域で使う汎用コンポーネント（`src/components`）と、各機能ドメインごとに閉じたモジュール（`src/features/ドメイン名/{components, api, hooks}`）を分ける構成。
   - **Pros:** 関連するUIコンポーネント、フック、API通信が1つのフォルダにまとまっているため、直感的で迷いが少ない（スピードとオンボーディングの優位性）。機能ごとに独立しているため、AIに対して「この機能のテストを書いて」とコンテキストを渡しやすく、初心者でも小さくテストを始めやすい（テスト容易性）。
   - **Cons:** FSDのようなLinterを用いた厳密な依存関係の強制力はないため、開発者の裁量に依存する部分がある。

3. **Next.js App Router デフォルト構成 (機能分割なし)**
   - **概要:** 階層分けを行わず、`app` ディレクトリや単一の `components` ディレクトリにすべてを配置する。
   - **Pros:** 環境構築が最速。
   - **Cons:** アプリケーションが成長した際にロジックとUIが散逸し、どこに影響範囲が及ぶか分からないためテストが非常に困難になる。保守性が著しく低い。

### Decision (決定事項)

**「軽量 Feature-Driven 構成」を採用する。**

バックエンドでは「個人開発の利点を活かした意図的な学習・ペインの体感」としてオニオンアーキテクチャを採用したが、プロジェクト全体としてフロントエンドにもオーバーエンジニアリングを導入すると、プロトタイプフェーズにおける開発速度が破綻するリスクが高い。
「フロントエンドテスト未経験」という自身の現在地と、「洗練されたUI/UXにリソースを全振りしたい」という要求に最も素直にフィットするのは、**複雑な階層ルールを排除しつつ関心の分離だけを担保できる軽量なFeature-Driven構成**であると判断した。

### Consequences (結果・影響)

- **Positive (メリット):** ディレクトリ構成で悩む時間（認知負荷）が最小化され、UI/UXの作り込みと、フロントエンドテスト（Vitest/React Testing Library等の基礎）の学習に十分な時間を確保できる。機能単位でファイルがまとまっているため、AIコーディングでの文脈生成が容易になり、品質の自動担保も進めやすい。
- **Negative (デメリット):** ディレクトリ間のクロスリファレンス（例: roadmap機能の中からactivity機能の内部層を直接インポートしてしまう等の違反）に対する強制的なブロック機構がない。これを防ぐためには、eslint-plugin-boundaries 等の簡易的な静的解析を後から導入するか、コードレビュー時の自主的なチェックが必要になる。
