# 仕様駆動開発ツール選定の結論

## 結論

本プロジェクトでは、**Claude Code等の任意のAIエージェントを使用した仕様駆動開発**を採用する。

spec-kitやKiroといった専用ツールではなく、AIエージェント単体での開発が最も合理的である理由は以下の通りである。


## 選定理由

### 1. コスト面での優位性

- **料金**: 専用IDEであるKiroはクレジット消費が不透明で、継続的な開発には月額$40〜200が必要となる。一方、Claude CodeはPro $20/月から利用可能であり、コスト予測も比較的容易である
- **移行コスト**: AIエージェント単体であれば、仕様書は標準的なMarkdown形式で管理するため、将来的に別のエージェント（Copilot、Gemini CLI等）への移行も容易である
- **導入コスト**: spec-kitは4段階のワークフロー習得、KiroはEARS記法と専用IDEへの適応が必要だが、AIエージェント単体であれば既存の開発環境をそのまま使用できる

### 2. ベンダーロックインの回避

- 仕様書を汎用的なMarkdown形式で管理することで、特定のツールやプラットフォームへの依存を排除できる
- AIエージェントの選択肢を将来にわたって維持でき、より優れたモデルやツールが登場した際に柔軟に乗り換え可能である

### 3. 仕様書管理の柔軟性

- 本プロジェクトは小規模開発でスピード重視であり、ユーザーストーリーと技術詳細の2種類の仕様書で十分である
- spec-kitの4段階プロセス（Constitution → Spec → Plan → Tasks）やKiroの3段階プロセスは、現時点ではオーバースペックである
- 必要な仕様書の粒度や形式をプロジェクトの実態に合わせて自由に調整できる

### 4. 将来のチーム拡大への対応

- 仕様書が標準的なMarkdown形式であれば、新規メンバーは特別なツール習得なしに内容を理解できる
- 将来的にチームが拡大し、より構造化されたワークフローが必要になった段階で、spec-kit等の導入を検討すれば良い
- メンバーごとに異なるAIエージェント（Claude Code、Copilot、Cursor等）を使用することも可能である


## 仕様書フォーマットの課題と対応方針

AIエージェント単体での開発には、標準化された仕様書フォーマットが存在しないという課題がある。この課題に対しては以下の方針で対応する。

### 対応方針

1. **既存フォーマットの参考**: cc-sdd、claude-code-spec-workflow等のコミュニティツールや、spec-kit/KiroのEARS記法を参考に、プロジェクトに適した形式を検討する

2. **ミニマルスタート**: 初期段階ではユーザーストーリーと技術詳細の2ファイル構成から始め、過度な形式化を避ける

3. **開発を通じた改善**: 実際の開発を進める中で「毎回記載している項目」「記載漏れしやすい項目」を特定し、テンプレートとして徐々に形式化していく

4. **継続的な見直し**: プロジェクトの成長やチーム拡大に応じて、必要であればより構造化されたフォーマットやツールの導入を検討する


## まとめ

小規模・スピード重視の本プロジェクトにおいては、専用ツールの導入コストやロックインリスクを負うよりも、AIエージェント単体で柔軟に開発を進め、仕様書フォーマットは実践を通じて最適化していくアプローチが最も合理的である。
