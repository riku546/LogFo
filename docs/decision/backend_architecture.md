# Architecture Decision Record: バックエンドアーキテクチャへのオニオンアーキテクチャの採用

**タイトル:** バックエンドアーキテクチャへのオニオンアーキテクチャの採用
**ステータス:** Accepted

### Context (背景と要件)

LogFoのバックエンドはプロトタイプフェーズであり、高速な開発スピードが求められている。一方で、AIコーディング（SDD）を前提としているため、テストを自動生成・実行できる「テスト容易性」が必要になる。また、将来的なチーム拡大を見据え、新規参画メンバーの「学習コスト」を抑える必要もある。しかし同時に、本プロジェクトは個人開発であり、開発者自身の深い技術的インサイトの獲得（学習意欲）も重要な目的の一つである。

アーキテクチャ選定にあたり、以下の4つの要素を評価軸とした：

1. **スピード:** プロトタイプフェーズにおける初期開発の速度。
2. **テスト容易性 (SDD):** ドメインロジックの単体テストの書きやすさ。
3. **学習コスト:** 将来のメンバーのオンボーディングのしやすさ。
4. **学習意欲:** 個人開発の利点を活かした、開発者自身の技術的挑戦。

### Options Considered (検討した選択肢)

要件を満たすためのアーキテクチャとして、以下の3つの選択肢を比較検討した。

1. **レイヤードアーキテクチャ **
   - **概要:** 機能別のディレクトリ内にController, Service, Repositoryを配置
   - **Pros:** ディレクトリが浅く直感的なため、開発スピードが最速。学習コストも低い。
   - **Cons:** オニオンやクリーンアーキテクチャの厳密な恩恵（ドメインの完全な隔離）を、システム全体で強制・体感するには少し弱い。

2. **オニオンアーキテクチャ (Onion Architecture)**
   - **概要:** Core（ドメイン/ユースケース）を中心に配置し、Infrastructure（DB等）やPresentation（UI/API）を外周層に完全に分断する構成。
   - **Pros:** ドメイン層が外部フレームワークやDBから完全に隔離されるため、SDD（AIによるテスト生成）と極めて相性が良い。依存関係逆転のルールの強力さを実践レベルで深く学べる。
   - **Cons:** ディレクトリの階層が深くなり、インターフェース等のボイラープレートが増えるため、初期スピードと学習コストの面では「レイヤード」に劣る。

3. **クリーンアーキテクチャ (Clean Architecture)**
   - **概要:** オニオンと概念は似ているが、層間のデータ変換（DTO等）のルールがより厳格な構成。
   - **Pros:** 完全無欠の独立性とテスト容易性。
   - **Cons:** ファイル数やマッピング処理が爆発的に増え、プロトタイプフェーズのスピード要件と致命的に合致しない（明らかなオーバースペック）。

### Decision (決定事項)

**「オニオンアーキテクチャ」を採用する。**

スピードと学習コストのみを合理的に事業視点で考慮すれば、「レイヤードアーキテクチャ」が最適解であった。しかし、本プロジェクトが「個人開発」であることを最大限の強みとして活かし、あえて学習曲線の急なオニオンアーキテクチャを選択した。これは無自覚なオーバーエンジニアリングではなく、「最適解である『レイヤード』との実装レベルの差分・ペイン」を実際に痛みを伴って体感し、将来の大規模開発の技術選定に活かすための「意図的な自己投資」である。

### Consequences (結果・影響)

- **Positive (メリット):** ドメインロジックがインフラから強制的に隔離されるため、AIを用いたテストコード生成（SDD）の精度と成功率が飛躍的に高まる。また、高度なアーキテクチャの運用知見という極めて強い個人の技術的アセットが手に入る。
- **Negative (デメリット):** ファイル単位の記述量（インターフェース定義や初期化処理など）が増加するため、初期の開発スピードは局所的に低下する。また、将来メンバーが参画する際、依存関係ルールのキャッチアップコストが発生する（ドキュメントでのカバーが必須となる）。


### Reflection (開発を通じた事後考察)
結論：オニオンアーキテクチャーの恩恵をあまり感じられなかった。
その原因は以下の２つになると考えています。

- 1 「DBを変える必要性がほとんどない」について
個人開発ではDB、ORMの移行はほとんど起きないため、DIPによってリポジトリー層の変更容易性が得られなかった。

- 2 「BtoB SaaSほどの複雑なドメインがない」について
個人開発ではBtoB SaaSほどの複雑なドメインがないため、ドメイン層を隔離する必要性が薄い。

しかし、オニオンアーキテクチャを採用したことによるメリットもあったと考えています。
- テスト容易性
usecase層を隔離したので、モックを使いやすく、テストが書きやすかった。
今回は仕様駆動開発（SDD）を採用していたので、仕様をテストコード落とし込むことで開発効率とLLMのハルシネーションを抑制できた。


### 開発時の苦労した点

レイヤードアーキテクチャであれば3ステップ程度で完了する修正が、オニオンでは以下の6ステップに倍増した：

Domain: Zodスキーマおよびエンティティ型の修正

Application (Interface): Repositoryインターフェースの引数・戻り値の型修正

Application (UseCase): ビジネスロジック内でのプロパティ処理の追加

Infrastructure (Database): Drizzleスキーマへのカラム追加

Infrastructure (Repository): インターフェースを満たすための実装修正およびデータの詰め替え（マッピング）

Presentation: Honoのルーティングおよびバリデーション層の修正

インターフェースの修正と各層のスキーマの整合性を手動で保ち続ける作業は、LLMによる自動生成という前提がなければ、開発スピードを下げるものであると実感した。
